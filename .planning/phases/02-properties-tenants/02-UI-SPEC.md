## Color

shadcn zinc theme provides the full palette. The contract below maps the 60/30/10 split to zinc semantic tokens.

| Role | shadcn Token | Usage |
|------|-------------|-------|
| Dominant (60%) | `background` (zinc-50 light / zinc-950 dark) | Page background, table row background |
| Secondary (30%) | `card` / `muted` (zinc-100 light / zinc-900 dark) | Property cards, tenant sub-rows, sidebar if added, table header row |
| Accent (10%) | `primary` (zinc-900 light / zinc-50 dark) | Reserved for specific elements listed below |
| Destructive | `destructive` (red-600) | Delete buttons and destructive confirmation dialogs only |

Accent reserved for:
1. Primary action buttons ("Add Property", "Add Tenant", "Add property", "Update property", "Add tenant", "Update tenant")
2. Active navigation link (current route indicator)
3. Focus ring on interactive elements (form inputs, table row on keyboard focus)

Accent is NOT applied to: table row hover, informational badges, column sort arrows, secondary buttons.

**Primary visual anchor:** The page heading ("Properties") paired with the "Add Property" primary CTA button at the top of the page is the primary focal point — the highest-contrast element pair that draws the eye first on page load.

## Interaction Patterns

### Form Pattern: Modal Dialog (locked decision)

All create and edit operations use a Dialog overlay. The underlying list stays visible and interactive is paused (modal blocks interaction while open).

- Dialog width: 480px on desktop, full-width on mobile
- Dialog opens centered in viewport
- Pressing Escape or clicking the backdrop closes the dialog and discards unsaved changes (no confirmation needed — shadcn Dialog default)
- The form submit button submits the form; on success the dialog closes and the list refreshes (see Copywriting Contract for exact button labels per dialog)
- Errors surface inline below the relevant field (react-hook-form + zod validation)

### Table Density: Comfortable (locked decision)

- Row height: 52px
- Cell padding: 16px horizontal, 14px vertical
- Column headers: 14px label weight, `muted-foreground` color, not bold
- Alternating row background: none — use hover state only (`muted` on hover)
- Actions column: right-aligned, contains Edit (pencil icon) and Delete (trash icon) icon buttons, 32px width each, 44px touch target
- Icon-only action buttons must declare explicit aria-labels: `aria-label="Edit property"`, `aria-label="Delete property"` on the Properties table; `aria-label="Edit tenant"`, `aria-label="Delete tenant"` on the Tenants table

### Delete Flow

Properties (PROP-03) and Tenants (TENT-04) require cascade warnings.

1. User clicks Delete icon button in table row
2. Alert Dialog opens: title "Delete [Property/Tenant]?", body explains cascade consequence (see Copywriting Contract)
3. Two buttons: "Keep property" or "Keep tenant" (outline) and "Delete property" or "Delete tenant" (destructive variant)
4. Confirming fires the delete API call; success closes dialog and removes row from table

## Layout Structure

The Phase 2 layout follows the scaffold's existing shell (Layout.jsx with top nav). Within that shell:

```
/properties
  PropertiesPage
    └─ Page heading + Add Property button (flex row, space-between)
    └─ Table: Address | Notes | Actions
    └─ [Dialog: Add/Edit Property form]
    └─ [Alert Dialog: Delete Property confirmation]

/properties/:id   (or inline expand — planner decides)
  PropertyDetailPage
    └─ Property header (address, notes, breadcrumb back to /properties)
    └─ Separator
    └─ Section heading "Tenants" + Add Tenant button
    └─ Table: Name | Monthly Rent | Due Day | Grace Period | Lease Dates | Actions
    └─ [Dialog: Add/Edit Tenant form]
    └─ [Alert Dialog: Delete Tenant confirmation]
```

The planner may choose to implement tenants inline below each property row or on a separate route. Either is consistent with this contract. The modal dialog pattern and table structure apply in both cases.
